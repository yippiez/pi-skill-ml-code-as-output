# Adding a new trainer

A **trainer** is one self-contained script in [ml/](ml/) named `train_<name>.py`.
It learns something from data and **prints a Python module to stdout**; you
redirect that into `next-pred` where the engine imports it. There is no registry
and no framework — a trainer is just a script with a `main()`. To add one, copy
the closest existing script and adapt it.

The 64 scripts already in `ml/` are your library of worked examples — almost any
new trainer is a small edit of one of them. Find the nearest match in
[catalog.md](catalog.md) (each catalog entry `<name>` is `ml/train_<name>.py`),
read it, copy it, change the dataset/fit/emit.

## Anatomy of a trainer

Every script follows the same four-part shape. This is `ml/train_c5.py`, the
canonical example (a C4.5/C5.0 decision tree for CBT candidate scoring):

```python
        '"""C4.5/C5.0-style decision tree — AUTOGENERATED.\n'
        f"Feature order: ({', '.join(map(repr, FEATURE_NAMES))},)\n"
        '"""\n\n'

        "def score(features):\n"
        "    raw = _raw_score(list(features))\n"
        "    return raw[1] if isinstance(raw, list) and len(raw) == 2 else float(raw)\n"
    )
    sys.stdout.write(out)       # …and print it to stdout
    return 0


if __name__ == "__main__":
    sys.exit(main())
```

The four parts, in order:

1. **Module-level imports + caps.** Only light imports at module scope: the
   project's feature/primitive helpers (`next_pred.engines.primitives.*`) and the
   dataset loader (`evals._hf.load_rows`). Heavy ML libs do **not** go here.
   Capping constants (`_MAX_TRAIN_ROWS`, depth, top-N) live up top so they're easy
   to find. See [dataset.md](dataset.md) for the data-loading conventions.
2. **`_dataset()`** — pull rows, filter to well-formed examples, featurize, return
   `X, y` (or whatever your fit step consumes). Stream and cap; these are small
   models by design.
3. **`main() -> int`** — **lazy-import every heavy dependency inside this
   function** (`sklearn`, `sympy`, `m2cgen`, a solver…). Fit/synthesize with a
   fixed seed. Keep it to seconds-to-minutes.
4. **Emit** — build the output module as a string and `sys.stdout.write` it (or a
   series of `print()`s, as the n-gram scripts do). The string starts with the
   autogen header + docstring, then the learned structure as literals, then a tiny
   pure function over them. The emitted module imports **only the stdlib**. Full
   conventions in [codegen.md](codegen.md).

Close with `if __name__ == "__main__": sys.exit(main())`.

## Step by step

1. **Pick the output shape and method.** Decide what the engine will import and
   call — a lookup `dict`, a `score(features)` fn, nested `if/else`, a grammar.
   Choose the catalog family whose output shape matches, then the specific method
   whose assumptions fit your data. See [catalog.md](catalog.md).
2. **Copy the nearest script.** `cp ml/train_<closest>.py ml/train_<yourname>.py`.
   Reuse its dataset loader, featurizer, and emit scaffolding — most of the work
   is already there.
3. **Swap the dataset.** Point `_dataset()` at your rows and featurizer. Keep the
   caps. If you need a HuggingFace dataset, load it through `evals._hf.load_rows`
   (see [dataset.md](dataset.md)).
4. **Swap the fit.** Replace the estimator/synthesis in `main()`. Keep heavy
   imports lazy and inside `main()`. Fix the seed.
5. **Swap the emit.** Make the output string serialize *your* learned structure as
   stdlib-only literals, rounding floats and capping to top-N so the diff is
   reviewable. Keep the autogen header. See [codegen.md](codegen.md).
6. **Run it and redirect into next-pred:**

   ```bash
   uv run python .pi/skills/ml-code-as-output/ml/train_<yourname>.py \
     > packages/knowledge/src/knowledge/learned/<name>.py
   ```

   Tables go in `tables/`, scorers/routers/formulas in `learned/`. Progress logs
   go to stderr (`file=sys.stderr`) so they never pollute the redirected stdout.
7. **Verify the artifact.** Import-check and lint the *emitted* file:

   ```bash
   uv run python -c "import knowledge.learned.<name>"
   ./scripts/lint.sh
   ```

8. **Add a catalog entry.** Add a one-line bullet under the right family in
   [catalog.md](catalog.md) so the next person finds it; include the
   `**Target:**` path if it emits to a fixed location.

## Heavy dependencies

A new method usually needs a library the workspace doesn't have yet (a solver, an
ILP engine, `pysr`, …). Add it with uv rather than hand-editing any
`pyproject.toml`, then lazy-import it inside `main()`:

```bash
uv add --optional ml <pkg>      # or the dev group
```

The split is deliberate: the *trainer* may depend on anything; the *emitted
artifact* must import only the stdlib. If your output file still needs the heavy
lib at runtime, you serialized the wrong thing — bake the learned numbers in as
literals instead.

## The odd one out: build_html_report.py

`ml/build_html_report.py` is not a trainer — it renders every code-as-output
technique into a single HTML page. Its `ROOT` is computed as
`Path(__file__).resolve().parent.parent`, which assumed the old repo-root `ml/`
location; re-point that constant before running it from inside the skill dir.

## Two things to keep straight

- **The script is the trainer; the committed `.py` under `next_pred` is its
  output.** The script is rerunnable to refresh the artifact — same task, same
  file. The engine never imports the trainer, only its output.
- **One script per technique, one artifact per task.** Don't generalize a script
  to cover many models; copy it. The point is that each `train_<name>.py` reads
  top-to-bottom as the recipe for exactly one artifact.
