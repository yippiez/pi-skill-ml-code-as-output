# Emitting the artifact

How to serialize a learned thing into a Python file that next-pred can import.

## Where artifacts live

Resolve targets relative to the repo root and keep them inside next-pred:

- **Lookup / successor tables** → `next-pred/src/next_pred/tables/<name>.py`
- **Learned scorers, routers, formulas** → `packages/knowledge/src/knowledge/learned/<name>.py`

Worked examples already in the tree: `learned/cbt_tree.py`,
`learned/nq_majority.py`, `learned/nq_train_qas.py`, `learned/wsc_tree.py`.

## The autogen marker

Every emitted module's **docstring ends in `— AUTOGENERATED`**, so a reader (and
grep) can tell at a glance the file is machine-produced and shouldn't be
hand-edited. Name the producing script where it helps:

```python
"""Per-question-type NQ majority answers — AUTOGENERATED."""
```
```python
"""SAT-minimal DFA — AUTOGENERATED via ``ml/train_dfa_identify.py``."""
```

The docstring states what the model predicts and, where relevant, the feature
order or data source — e.g. `Feature order: (...)` for a scorer fed a feature
vector. This is the convention the restored `ml/train_*.py` scripts already
follow; match it rather than inventing a comment header.

## Module shape

A clean emitted module is: docstring → (stdlib imports only) → the learned
structure as literals → a tiny pure function over them. Keep the layout regular:
docstring first, imports block separated by one blank line, each body statement
separated by a blank line, trailing newline.

```python
"""<what this predicts>. AUTOGENERATED — see header."""

CHAR_BIGRAM_SUCC: dict[str, str] = {
    'a': 'n',
    't': 'h',
    # …top-N, sorted, capped…
}

def successor(c: str) -> str | None:
    return CHAR_BIGRAM_SUCC.get(c)
```

### Emitting literals

- Big dicts/lists: one item per line, `f"    {k!r}: {v!r},"`, keys sorted, with
  a `NAME: <type_hint> = {` opener and `}` closer. Cap to top-N by frequency and
  round floats so the diff is reviewable.
- Small functions: a `def name(args):` with a single `return` expression is
  usually enough — push the learned content into module-level literals the
  function reads, not into the function body.

## Fit-time conventions

- **Lazy-import heavy deps inside the fit step**, never at module top level:
  `from sklearn.naive_bayes import GaussianNB` goes inside the function that
  trains. The skill must not require an install to be loaded.
- **Cap the dataset.** Stream, take the first N rows, filter to well-formed
  examples. These are small models by design.
- **Reproducible + fast.** Fix any seed, keep the fit to seconds-to-minutes.
- The *emitted* file must import only the stdlib (or nothing). If your model
  needs `math`, that's fine; if it needs `sklearn` at runtime, you emitted the
  wrong thing — bake the learned numbers in as literals instead.

## Running a trainer

A trainer **prints the emitted module to stdout**; you redirect it into the
target file. Progress logs go to stderr (`print(..., file=sys.stderr)`) so the
redirected stream stays clean module source.

```bash
uv run python .pi/skills/ml-code-as-output/ml/train_<name>.py \
  > packages/knowledge/src/knowledge/learned/<name>.py
```

## After emitting

1. Redirect the script's stdout to the target file (create parent dirs as
   needed; `tables/` for lookup tables, `learned/` for scorers/routers/formulas).
2. Import-check it: `uv run python -c "import next_pred.<...>.<name>"`.
3. Run `./scripts/lint.sh` and fix anything it surfaces (it auto-fixes; use
   `--check` to dry-run).
